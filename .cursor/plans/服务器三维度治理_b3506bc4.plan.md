# RPG Server 三维度整体改造计划

## 目标与范围

围绕以下三个目标推进长期治理：
- **稳定性**：消除单线程事件循环阻塞与级联故障，建立统一超时/重试/背压策略
- **安全性**：补齐边界防护与身份可信链，收敛凭据与证书风险，形成持续安全治理
- **扩展性**：拆除控制面单点、修复扩容闭环，逐步实现 AOI/Record 可弹性扩展

## 风险优先级（先后顺序）

- **P0（立即）**
  - `Session` 同步关系加载阻塞事件循环（会话服冻结风险）
  - `Record/Login` 在 handler 内同步 MySQL（高并发下全进程阻塞）
  - `Super` 控制面单点（注册/编排/外联汇聚）
  - 登录与鉴权缺少速率限制

- **P1（短中期）**
  - 登录链路超时预算不一致（12s/15s/17s）
  - 发送背压失败处理不一致（部分 `SendMsg` 失败未闭环）
  - Lua 默认全库开放，缺少沙箱边界
  - 区内 mTLS 证书复用导致“持证即高信任”

- **P2（中长期）**
  - Scene/Session 负载指标闭环缺失（`playerCount` 驱动弱）
  - AOI 单进程不可水平扩展
  - InternalMsg 定长结构演进成本高、缺版本治理

## 分阶段路线（12 个月）

### 阶段 1：止血与基线（0-3 个月）

- 稳定性
  - 将 `Session` 关系加载改为异步状态机，移除 handler 内长轮询
  - 统一登录链路超时预算，集中在 [sdk/util/LoginFlowTimeouts.h](/home/hcg/RPG_Server/sdk/util/LoginFlowTimeouts.h)
  - 统一 `SendMsg` 失败语义（告警 + 重试或显式失败返回）
- 安全性
  - 强制生产 `TLS enabled=1`、`verifyPeer=1`，禁止弱配置上线
  - Gateway/Login 增加按连接/IP/账号的速率限制
  - Scene 入口补齐基于 `clientConn` 的二次一致性校验
- 扩展性
  - 补齐 Session/Scene 负载指标上报，让选服策略真正生效
- 观测
  - 建立登录链路 SLI：成功率、P95/P99、pending 深度、超时分布

### 阶段 2：控制面减负（3-6 个月）

- 稳定性
  - 统一内外联重连器：指数退避、熔断、半开检测
  - 子服断连时对 pending 事务做立即失败/重排队策略
- 安全性
  - 每进程独立证书与身份映射（证书主体 -> serverType/serverId）
  - Login 引入与 Gateway 同级的客户端消息 Validator
- 扩展性
  - 弱化 Super 热路径编排职能，聚焦注册/路由/健康管理
  - 服务发现增量刷新，不再只靠启动期 `ServerList`

### 阶段 3：计算与存储弹性（6-9 个月）

- 稳定性
  - Record 持久化改有界队列与写合并，降低主循环阻塞
- 安全性
  - 凭据与证书轮换自动化（短周期、可审计）
- 扩展性
  - AOI 按 map 维度分片（多 AOI 实例）
  - Scene 对 AOI/Session 连接从单点改为池化与一致性路由
  - CharBase/Relation 开始垂直拆分与按需加载

### 阶段 4：平台化与跨区（9-12 个月）

- 稳定性
  - Super 高可用（多实例 + 共享注册状态）
- 安全性
  - 安全治理常态化：SAST/依赖审计/演练机制
- 扩展性
  - Zone/Global 从骨架走向跨区业务总线
  - InternalMsg 引入版本化兼容策略（逐步替代强依赖定长结构）

## 落地机制

- **变更门禁**
  - 每阶段必须有：可观测指标、灰度开关、回滚路径、压测结果
- **发布策略**
  - 先单服灰度，再按 serverId 扩容；涉及协议变更时采用双栈期
- **验收标准**
  - 稳定性：登录成功率、超时率、重连风暴次数显著下降
  - 安全性：高风险项（P0/P1）闭环率与误配率下降
  - 扩展性：新增 Scene/AOI/Record 实例可在不停服条件下接入
