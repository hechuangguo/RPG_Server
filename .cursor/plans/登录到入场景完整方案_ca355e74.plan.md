# 账号登录到进入场景技术方案

## 目标

在现有“区列表 + 注册登录入口”基础上，补齐从点击登录到角色进图的完整闭环，满足以下约束：

- 账号可拉取角色列表并选择角色进入游戏。
- 登录链路必须校验区服状态、账号安全、重复登录、网关/场景负载。
- 由 SuperServer 统一编排登录流程与跨服协调。
- 首次进入主城新手村出生点，失败可回退，链路可观测。

- LoginServer 已支持区列表、注册、账号密码校验入口（文档输入）。
- 网关/区内服已有 S2S 注册与心跳体系（见项目约束）。
- SessionServer 负责场景/副本分配，SceneServer 负责地图实体。

## 详细分段设计

### 1) 登录鉴权与票据（LoginServer）

- 客户端提交账号/密码/zoneId。
- LoginServer 校验：账号存在、密码哈希、账号锁定状态、zone 可用。
- 登录成功后生成短期一次性登录票据 `loginTicket`（带 accountId、zoneId、过期时间、nonce、签名）。
- 返回客户端：`loginTicket` + 下一跳 SuperServer 接入信息。

建议：

- 票据 TTL 30~60 秒；单次使用后立刻失效。
- 记录设备指纹/IP 变化，触发风险等级（仅记录或挑战）。

### 2) 角色列表与角色选择

- 客户端携带 `loginTicket` 请求角色列表。
- SuperServer 向 LoginServer 验票后，调用 RecordServer 拉取该账号下角色列表。
- 客户端选择角色后提交 `roleId` + `loginTicket` + `clientNonce`。

校验点：

- `roleId` 必须属于该账号。
- `roleId` 不得处于“登录中”或“在线中”（见并发控制）。

### 3) SuperServer 登录编排（核心）

SuperServer 作为编排器，顺序执行：

1. 验票（LoginServer）。
2. 重复登录互斥加锁（按 roleId/accountId）。
3. 选网关（Gateway 负载均衡 + 可用性）。
4. 载入角色（RecordServer）。
5. 选场景（SessionServer 按 mapId/负载）。
6. 建立绑定（roleId -> gatewayConnId -> sceneServerId）。
7. 下发进入场景命令并等待确认。
8. 成功后返回客户端进场信息；失败则按回滚策略撤销状态。

### 4) 负载均衡策略

网关均衡（Gateway）：

- 过滤不健康节点（心跳超时、维护中）。
- 评分：`score = connCount * a + recentFail * b + rtt * c`，取最小。
- 增加同账号短时间粘性，减少切换抖动。

场景均衡（Scene）：

- SessionServer 基于 mapId 找可承载 SceneServer。
- 评分包含在线人数、单位时间入场量、实例压力。
- 首次进图默认主城新手村坐标；拥挤时可分流至同地图不同实例。

### 5) 重复登录与并发安全

状态机（按角色）：

- `OFFLINE -> LOGIN_PREPARE -> ENTERING_SCENE -> ONLINE`
- 任何失败回退到 `OFFLINE`。

并发规则：

- 同一 `roleId` 同时仅允许一个登录事务。
- 同一账号多端登录策略可配置：踢老会话或拒绝新会话。
- 锁超时自动释放，防止僵尸状态。

### 6) 失败回滚与幂等

关键失败点与处理：

- 验票失败：立即拒绝，记录安全日志。
- Record 拉角色失败：释放锁，返回可重试错误码。
- Scene 分配失败：释放网关预占，释放锁，提示排队或稍后重试。
- 网关绑定后 Scene 失败：执行解绑补偿，角色状态回退。

幂等：

- 使用 `loginTxnId`（客户端或 Super 生成）贯穿整个链路。
- 对重复请求返回同一事务结果，避免重复进图。

### 7) 出生点与首次入图

- 若角色无有效位置或首次创建：
  - `mapId = mainCityMapId`
  - `pos = newbieSpawnPos`
- 地图/坐标从配置表加载，禁止硬编码魔法数。

配置建议：

- 主城地图与出生点配置放在静态配置（DataDoc -> database Lua）并由 Scene 加载。

### 8) 安全与风控

- 账号密码只在 LoginServer 验证，不向区内服传播明文或哈希。
- 票据签名 + 过期 + 一次性消费，防重放。
- 登录限流：按 IP、账号、设备维度。
- 审计日志：每次登录事务输出 `txnId/accountId/roleId/zoneId/gatewayId/sceneId/code`。

## 分期实施

### Phase 1（最小闭环）

- 票据登录 + 角色列表 + 角色选择进入。
- Super 串行编排 + 基础回滚。
- 固定主城出生点。

### Phase 2（稳定性增强）

- 完整并发锁与幂等事务。
- 网关/场景评分负载均衡。
- 登录全链路超时和重试策略。

### Phase 3（安全与运营）

- 风控限流与异常检测。
- 完整观测（日志、指标、告警）。
- 登录/进图性能优化（热点缓存、批量读取）。

## 验收标准

- 功能：账号能稳定完成“登录 -> 选角 -> 入图”。
- 安全：票据不可重放，重复登录策略生效。
- 稳定：超时/失败能回滚，无角色卡死 `ENTERING_SCENE`。
- 性能：高并发下登录耗时、错误率、重试率在阈值内。

## 风险与规避

- 事务跨服导致状态不一致：用 Super 事务号 + 显式补偿。
- Scene 瞬时过载：增加准入阈值与排队提示。
- 协议扩展影响兼容：模块化新增 sub，保留旧流程开关。
